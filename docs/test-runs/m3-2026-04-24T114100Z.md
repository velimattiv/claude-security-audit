## Real Juice Shop bugs the bundle caught

Quick sample of confirmed issues from the dogfood run:

- **Hardcoded JWT secret** (`semgrep:hardcoded-jwt-secret`, in the repo's
  JWT signing code) — real issue, classic Juice Shop challenge.
- **Sequelize injection** (`semgrep:express-sequelize-injection`, 6 hits) —
  deliberately injectable endpoints Juice Shop uses to teach SQLi.
- **50 secrets in fixtures** (`gitleaks:generic-api-key` etc.) — mostly in
  `data/static/users.yml` where the CTF user accounts live with 2FA
  secrets. Expected finding; severity assignment in M5 should de-prioritize
  fixture-file findings.
- **Open redirect** (`semgrep:express-open-redirect`) — another classic
  Juice Shop challenge.

None of these should be surprising — Juice Shop is a textbook target.
What matters is: **the scanner plumbing works and the slim-SARIF output is
consumable by Phase 7 synthesis without mutation**.

## Corrections to spec docs (applied during M3)

1. `phase-04-scanners.md` invocation for osv-scanner said
   `osv-scanner scan source --recursive ...`. OSV 1.9.x uses positional
   `[directory]` after `scan` — fixed to `osv-scanner scan --recursive
   --format sarif --output ... .`. Updated the doc + `scanner-bundle.md`.
2. `phase-04-scanners.md` said `semgrep --config auto`. In practice,
   `--config auto` requires `--metrics=on`, which violates the locked
   decision Q4 (no telemetry). Switched the documented default to explicit
   named rulesets (`p/security-audit`, `p/owasp-top-ten`, `p/jwt`) and
   documented the `auto` path as an opt-in alternate.
3. `lib/sarif-postprocess.md` referenced an 80% compression estimate with
   no evidence. Replaced with measured figures per scanner (82-99%) from
   this dogfood.

## Observations for later milestones

1. **Scanner install time is non-trivial.** Trufflehog's install.sh took
   ~5 min to download a 30MB tarball from GitHub on this host. Trivy's
   took even longer (~3 min for 40MB). First-run users should expect
   several minutes. The installer should print progress (it currently
   prints per-scanner status lines but not download progress bars). Minor
   UX improvement for M7.
2. **OSV-scanner without a lockfile is blind.** If the project doesn't
   commit its lockfile, SCA coverage is zero. Phase 4 summary should flag
   this explicitly (the dogfood's summary.json does). Consider adding a
   pre-Phase-4 step that runs `npm/pnpm/yarn/poetry install --frozen-lockfile`
   if a manifest is detected but no lockfile — or simply a warn-and-skip
   with a doc link. v2.1 candidate.
3. **Gitleaks on a shallow clone is degraded.** Working tree vs history
   scans returned the same 50 results because only 1 commit exists. The
   summary documents this. On a full clone, history scan would surface
   secrets that were added and later "removed" in a subsequent commit.
4. **Sub-agent vs direct Bash for scanner runs.** The spec says each
   scanner runs via a sub-agent for context isolation; the dogfood ran
   them directly via Bash to keep iteration fast. Both are correct; the
   sub-agent model is documented and will be exercised on the first real
   end-to-end run (M5+ dogfood).
5. **Built-in reviews deferred.** This M3 dogfood did not invoke
   `/security-review` or the vendored adversarial-review — those require
   LLM calls (sub-agents) and are downstream of scanner plumbing. They
   are validated in the M5 dogfood where Phase 7 synthesis needs their
   output.
