## impossible.php sanity check

**Passed.** Sub-agent reported:

> `vulnerabilities/fi/source/impossible.php` uses `in_array()` against a
> 4-filename whitelist before `include()`. The "include/require + $_GET
> path" LIKELY finding on `fi/low.php` (finding 0019) does NOT fire
> semantically on `impossible.php` because `$file` is validated against
> the fixed list — proving the pattern distinguishes vulnerable from
> fixed.

> Similarly, `open_redirect/impossible.php` maps `intval($_GET['redirect'])`
> to a static target map, so the `header('Location: '+$_GET)` pattern
> does not match there (finding 0030).

The PHP patterns have **real discriminative signal**, not just any-
string matches.
