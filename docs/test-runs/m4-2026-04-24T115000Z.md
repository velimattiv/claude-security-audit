# M4 Dogfood — Deep Dives on Juice Shop, 2026-04-24T11:50:00Z

**Target.** `/tmp/audit-targets/juice-shop` at `3b178fd` (shallow clone).
**Skill version.** `2.0.0-m4`.
**Scope.** Phase 5 (Parallel Deep Dives). Two of the nine categories
executed as real sub-agents against the Juice Shop partition — the rest
validated at the category-instruction level (spec exists, grep catalog
exists, gates exist) but deferred to M5's end-to-end dogfood for
execution.

## Why only 2 of 9 categories run in this dogfood

All 9 category instruction files ship in M4 (cat-01 through cat-09). The
dogfood protocol for each milestone is to validate the *new* mechanics
works on a real target — it is not a full production run. Phase 5's
fan-out logic calls for `partitions.full × 9` sub-agents at 8-concurrent
throttle; exercising 2 of them is sufficient to validate:
1. Sub-agent prompt + template round-trip.
2. JSONL finding schema is populated correctly by real Opus sub-agents.
3. Scanner cross-referencing (Phase 4 slim SARIF → Phase 5 CONFIRMED
   confidence) works.

Categories exercised:
- **cat-01 Auth & Authz** — high-yield on Juice Shop (missing auth is
  its signature vulnerability class).
- **cat-08 Injection / SSRF / Deserialization** — highest confirmed-by-
  semgrep count (6 Sequelize injection + 1 open redirect + XXE/SSRF/YAML).

## Sub-agent execution

Both sub-agents spawned in parallel via the `Agent` tool with
`subagent_type: general-purpose`, `model: opus`, following the prompt
template at `templates/subagent-prompt.md`. Each finished in ~6.5 min
(395s / 398s wall clock). Token usage: 150K-167K per sub-agent.

### cat-01 Auth & Authz sub-agent — 40 findings

| Severity | Count |
|---|---|
| CRITICAL | 3 |
| HIGH | 23 |
| MEDIUM | 12 |
| LOW | 2 |

Confidence: **4 CONFIRMED**, 36 LIKELY. (CONFIRMED require a scanner
corroboration — 4/40 is appropriate for a spot dogfood where scanner
coverage is partial.)

Top finding: `juice-shop:auth:0001` — *Hard-coded RSA private key used
to sign JWTs* (CWE-798, ASVS-V2.10.3, API2:2023). Sources: gitleaks
`private-key` + semgrep `hardcoded-jwt-secret` + manual
`jwt-private-key-constant`. The sub-agent correctly recognized that the
RSA key in `lib/insecurity.ts:23` is used by `security.authorize()` and
the matching public key is served at `/encryptionkeys/jwt.pub` — so
anyone reading the public Juice Shop repo can forge any user's JWT. Full
attack scenario and suggested fix included in the JSONL row.

manual. Attack scenario: `POST /rest/user/login` with
`{"email":"admin@juice-sh.op'--","password":"x"}` returns an admin JWT.

## Cross-reference with Phase 4 scanners

For each Phase 5 finding whose `(file, line)` overlaps a Phase 4 scanner
hit, the sub-agent set `confidence: CONFIRMED` and attached the scanner
source. Observed cross-links (from auth + injection):

| Scanner hit | Promoted to CONFIRMED |
|---|---|
| semgrep `express-sequelize-injection` | cat-08 SQLi findings (login handler, search) |
| semgrep `hardcoded-jwt-secret` | cat-01 auth:0001 hardcoded RSA key |
| gitleaks `private-key` | cat-01 auth:0001 (second source) |
| gitleaks `kekse` cookie-signer constant | cat-01 cookie-signer finding |

The cross-referencing mechanic works as designed: scanner-backed
findings carry CONFIRMED confidence; grep-only findings stay at LIKELY.

## Observations for later milestones

1. **Sub-agents respect the 40-cap cleanly.** Both prompts capped at
   40; both sub-agents returned exactly 40 findings and noted in
   `notes` what they dropped (auth: "8 extra niggles"; injection: full
   coverage of the main CWE families). In a real run (cap removed or
   raised), Juice Shop would likely yield 100-200 per category —
   something to bench on M5.
2. **Category file size is tractable.** Each `cat-<NN>` file weighs
   150-300 lines. Sub-agent context held comfortably.
3. **Scanner cross-reference is the biggest confidence lever.**
   Without Phase 4 output, every finding would default to LIKELY. With
   the slim SARIF in scope (82-99% reduction vs raw), 4 of 40 auth
   findings and ~6 of 40 injection findings promoted to CONFIRMED.
4. **Polyglot grep patterns weren't exercised.** Juice Shop is TS/JS
   only; the per-language patterns in cat-04 (MITM) and cat-05 (Crypto)
   for Go / Python / Java / Rust / etc. still need exercise. M5's
   end-to-end dogfood should include at least one non-Node target
   (DVWA for PHP, or a small Go repo with TLS misconfig) per the M3
   deferral note.
5. **Trivy vuln-DB dependence in slow networks.** Trivy's 91MB vuln DB
   download timed out on the container's constrained network. Trivy
   `config` (no DB needed) produced 6 findings cleanly. M7 README
   should document the `trivy fs --skip-db-update` workaround for
   offline / air-gapped environments.
6. **40-finding cap is arbitrary for production.** The cap was imposed
   by the dogfood prompt for predictability; the phase-05 spec doesn't
   cap. M5 synthesis may want a severity-weighted de-duplication
   (merge near-identical HIGHs; surface one per family) rather than a
   hard numeric cap.
7. **Finding IDs are clean.** Both sub-agents used the prescribed
   format `<partition>:<category>:<seq>` (e.g., `juice-shop:auth:0001`)
   with 4-digit zero-padding. Phase 7 deduplication will key on
   `(file, line, category, fingerprint)`, not `id`, so ID collisions
   across partitions are not a concern.
