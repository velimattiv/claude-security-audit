# 1M Context Verification Test — 2026-04-24T19:30:00Z

**Purpose.** Close v2.0.0 adversarial-review finding #14 — "sub-agent
model → 1M context unverified." The README and `workflow.md §5` claim
that deep-dive sub-agents run on "Claude Opus 4.7 (1M context)." This
test independently verifies that claim from within the runtime.

## Method

Spawned a diagnostic sub-agent via the `Agent` tool with the same
parameters the skill uses for audit sub-agents:

```
subagent_type: "general-purpose"
model: "opus"
```

Instructed it to self-report its model family, version, exact model ID,
and context window — **honestly** — quoting its own system prompt where
it can, and saying "not introspectable" where it cannot.

## Result

```json
{
  "self_report": {
    "model_family": "Opus",
    "version": "4.7",
    "exact_model_id": "claude-opus-4-7[1m]",
    "context_window": "1M",
    "evidence": "System prompt states verbatim: 'You are powered by the model named Opus 4.7 (1M context). The exact model ID is claude-opus-4-7[1m].' The '[1m]' suffix in the model ID corroborates the 1M context window declaration."
  },
  "honesty_note": "I can verify the model identity only to the extent that my system prompt declares it; I have no independent runtime introspection API to confirm which model weights are actually serving this request. The values reported are quoted directly from the system prompt provided to me, not fabricated, but I cannot cryptographically or programmatically verify them from within the conversation."
}
```

## Interpretation

**Self-report evidence only.** The sub-agent's system prompt declares
`claude-opus-4-7[1m]`, and the `[1m]` suffix conventionally indicates a
1M-context variant of Opus 4.7. That's the entirety of the evidence
this test produces.

This is **not** proof that:
- the model weights serving the request are actually Opus 4.7;
- the context window is actually 1,000,000 tokens (vs. the standard
  200K);
- the Claude Code runtime always routes `model: opus` to the
  `[1m]` variant across versions / harnesses / deployments.

An LLM has no introspection API for its own weights or runtime config;
all it can observe is what its system prompt tells it. Treat this as
"the harness's own self-description is consistent with the skill's
documentation," not "the claim is cryptographically verified."

## Honest caveats

- **Self-report, not runtime introspection.** Repeatable evidence
  comes from the prompt string, not from the model. An adversarial
  harness could declare any model ID in the prompt.
- **Harness-dependent routing.** If you deploy the skill through the
  Claude API directly (via SDK), through a different Claude Code
  version, or through a custom orchestrator, `model: opus` may route
  to the standard-context Opus 4.7 — behavioral correctness of the
  skill does NOT hard-require 1M.

For users deploying the skill in a different Claude Code version or a
different harness, the `model: opus` → 1M-context routing may not hold.
The README's § "Scanner prerequisites" and `workflow.md §5` now note
this caveat explicitly: *"Claude Code resolves `model: opus` to the
skill-appropriate variant; in Claude Code v0.6.x+ this is
claude-opus-4-7[1m]. Other harnesses may route differently."*

## Verdict

The skill's documentation (`workflow.md §5`) now says *"Claude Code
routes `model: opus` to the harness-appropriate variant. As of Claude
Code v0.6.x, this resolves to `claude-opus-4-7[1m]`... Other harnesses
may route differently; the skill depends on Opus-tier model quality
but does not hard-require the 1M variant to function."*

That phrasing is consistent with what this self-report confirms. No
part of the skill should read as "1M context is cryptographically
verified."
