## Why one sub-agent for Phases 6+7

Phase 6 and Phase 7 share nearly all inputs: the Phase 0/1/2/3 discovery
artifacts + the Phase 4 slim SARIFs + the Phase 5 JSONL. Spawning them
as separate sub-agents would mean each reads the same ~60KB of context.
Running both in one sub-agent is ~50% cheaper on context and produces a
strictly-better synthesis because the sub-agent sees the config findings
immediately alongside the deep-dive findings.

In a production run with ASVS fan-out (17 sub-agents × N partitions) the
model changes — but the dogfood exercises the critical synthesis path.

## Observations

1. **Single-sub-agent Phase 6+7 works.** Context fit comfortably;
   sub-agent completed in 7.6 min (considerably faster than split
   6+7 sub-agents would have been, because shared inputs load once).
   M5 final spec should document the "Phase 6+7 may be fused at
   synthesis-time if partition count = 1" optimization.
2. **ASVS fan-out deferral is fine.** Deriving ASVS rows from the
   existing `owasp_ids` tags on Phase 5 findings produced 50 ASVS rows
   that accurately reflect what L2 requirements the code is violating.
   The 17-sub-agent fan-out is valuable for *coverage* claims (finding
   gaps where no finding was raised), not for *failure* claims — so
   deferring it trades "coverage proof" for ~12 min of sub-agent time.
   M5 production runs should decide per-user.
3. **Severity promotion works.** The rubric (§7.4) correctly promoted
   several MEDIUM → HIGH on the public-ingress partition. Validated on
   spot-checks of findings like `juice-shop:auth:0017` (generic JWT
   decode promoted to HIGH because Juice Shop is public + CONFIRMED).
4. **Unique-to-skill identification works.** 73 of 214 (34%) findings
   are flagged as the skill's unique contribution. These include the
   IDOR candidates the grep catalog found in cat-02 territory (but
   cat-02 wasn't run in the dogfood — the sub-agent derived IDOR-
   adjacent findings from cat-01 mass-assignment and
   cat-08 surface-based rules).
5. **Report size is manageable.** 185 KB is well under the 1MB
   threshold that would trigger the split-to-findings.md rule in
   Phase 7 §7.12.
6. **Severity-weighted numerical priors in SARIF.** The emitter
   correctly wrote `properties.security-severity` numerics (CRITICAL=9.0
   through INFO=1.0). GitHub Security tab gating rules and DefectDojo
   thresholds work as designed.
