### Fingerprint correctness

Initial implementation used base64 truncation for `fingerprint`, which
produced identical prefixes for findings sharing the same file path
(e.g., two findings in `lib/insecurity.ts` both hashed to the base64
prefix of `"lib/insecuri"`). Switched to **sha1 of
`file:line:title`, truncated to 12 chars**. Validation: first three
fingerprints from the 80-finding carryover are now distinct:
`3ffa415c5a9b`, `f537dc129730`, `a0579304361f`.

Committed the fix and kept a note in the baseline schema's description.

## Observations for follow-up

### 1. Phase 2 surface `file` field — registration site vs handler body

The Phase 2 sub-agent (M2) recorded `file: "server.ts"` for every
Express `app.get/app.post/...` registration — Juice Shop's pattern
puts the registration in `server.ts` and the handler body in
`routes/<resource>.ts`. The `handler_hash` was computed against the
registration line (`app.post('/rest/user/login', login())`) rather
than the handler body in `routes/login.ts`.

**Impact on delta mode:** a change to `routes/login.ts` does NOT
invalidate any Phase 2 surface row via the `file ∈ changed_files`
rule. It does invalidate via the keystone-file rule (because
`routes/login.ts` is a keystone), so the net effect for Juice Shop is
correct — but the path through the algorithm is less direct than
intended.

**Fix** (M2 follow-up or v2.1): Phase 2 sub-agent should emit BOTH
`registration_file` (where `app.post` lives) and `handler_file` (where
the actual handler function body lives). The `handler_hash` should be
computed against the handler body. Filed as a v2.1 candidate in
`TODO.md`.

### 2. Single-partition repos see a degenerate carryover

For a one-partition repo, any touched partition forces a full
re-audit — no findings carry forward. The delta-mode savings kick in
only on multi-partition repos (monorepos, multi-service). This is by
design — spec §V2-SCOPE.md §5 is explicit that carryover is a
per-partition optimization — but document it clearly in the user-
facing README so single-partition users don't expect the 2-5 min
target.

### 3. Fingerprint stability across runs

The sha1 fingerprint depends on `file:line:title`. If the title drifts
between runs (e.g., the sub-agent phrases the same finding
differently), the fingerprint changes and the finding looks like a new
row. Mitigation for v2.1: fingerprint off `file:line:cwe:rule_short`
instead, so findings of the same CWE class at the same file+line are
stable even if wording drifts.

### 4. Pruned baseline size

53KB for a 103K-LOC monolithic repo. Scaling linearly, a 1M-LOC
monorepo with ~8 partitions might produce ~500KB pruned. Still
tractable for git. If it becomes an issue, a v2.1 compression step
could gzip the pruned file (git handles binary blobs fine with
`git lfs` if needed).

### 5. Dogfood didn't exercise the `adjacency` expansion

`lib/delta-mode.md §3` says: "Add adjacency: partitions that share a
keystone with a touched one." Juice Shop has one partition, so there
is no adjacency to expand. Adjacency is validated architecturally
(the spec is clear) but operationally untested. Flagged for the first
multi-partition dogfood.
