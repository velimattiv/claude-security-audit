# Phase 7: Synthesis & Report

**Goal:** Cross-reference ALL findings from all phases. Deduplicate. Assign final severities. Produce the structured audit report. Offer to create a GitHub issue.

## Actions

### 7.1 — Collect All Findings

Gather findings from:
- Phase 1: `/security-review` findings, adversarial review findings, dependency audit
- Phase 2: Route inventory immediate flags
- Phase 3: Authorization matrix findings (from agent)
- Phase 4: IDOR & data ownership findings (from agent)
- Phase 5: Token/PAT scope findings (from agent) — if applicable
- Phase 6: Security configuration findings

### 7.2 — Deduplicate and Cross-Reference

For each finding:
1. Check if the same issue was found by multiple sources (e.g., both `/security-review` and the auth matrix audit found the same missing auth check)
2. If found by multiple sources: **merge into a single finding with higher confidence**, note all sources
3. If found by only one source: keep as-is, note the single source

Mark findings with confidence level:
- **CONFIRMED** — found by 2+ independent sources, or directly verified in code
- **LIKELY** — found by 1 source with strong evidence in code
- **POSSIBLE** — found by 1 source, could not be fully verified (e.g., depends on runtime behavior)

### 7.3 — Assign Final Severity

Apply consistent severity using this rubric:

| Severity | Criteria |
|----------|----------|
| **CRITICAL** | Direct path to unauthorized data access, privilege escalation, or remote code execution. Exploitable without specialized knowledge. |
| **HIGH** | Auth bypass, IDOR, token scope escape, or security misconfiguration that exposes user data. Requires some knowledge to exploit. |
| **MEDIUM** | Information disclosure, missing security headers, weak configuration, or issues that require chaining with another vulnerability. |
| **LOW** | Defense-in-depth improvements, best practice deviations, or issues with minimal practical impact. |
| **INFO** | Observations and recommendations that are not vulnerabilities but could improve security posture. |

### 7.4 — Identify Unique Value

Explicitly call out findings that were **only found by the systematic audit** (Phases 2-5) and NOT by `/security-review` or adversarial review. This demonstrates the unique value of inventory-based auditing over generic scanning.

### 7.5 — Generate Report

Produce the report using the template below. Save to:
- `_bmad-output/implementation-artifacts/security-audit-report.md` (if a `_bmad-output/` directory already exists in the project — for users who also run BMAD alongside this skill)
- Otherwise: `docs/security-audit-report.md`

### 7.6 — Present to User and Offer Next Steps

After presenting the report, inform the user:

> **Next steps you can take:**
> - Ask me to **fix** any specific finding (e.g., "fix finding C1")
> - Ask me to **fix all** findings of a severity level (e.g., "fix all CRITICAL and HIGH findings")
> - Ask me to **create a GitHub issue** from this report (e.g., "create a GitHub issue for this security audit report")
> - Run `/security-audit` again after fixes to verify remediation

For the GitHub issue option: use `gh issue create` with the report content formatted for GitHub, applying the label `security` if it exists. Title format: `Security Audit: [count] findings ([critical] critical, [high] high) — [date]`. Mark the issue body with a `<details>` block for the full report to keep it scannable.
